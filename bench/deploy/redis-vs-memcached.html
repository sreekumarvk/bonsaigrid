How to read this

Fair by construction. Each server and the client run in a separate cgroup on a disjoint cpuset with equal CPU/memory caps, so throughput reflects the server — not the Linux scheduler arbitrating between a CPU-heavy client and the server for the same cores.

Verified every run. The loadgen compares every GET to the value just written; the Correctness tile must read 0 mismatches. This catches a server that acks writes without storing them (throughput and latency alone can't — a miss is cheap).

Same-client vs cross-client. BonsaiGrid and Hazelcast are driven by the same official Hazelcast client, so that pair is directly comparable; Redis and Memcached use their thin native clients, so cross-backend numbers are directional. The dashed line is BonsaiGrid's server driven by a thin raw-protocol client — its ceiling with the client tax removed.

Server saturation — sampled live from each server's cgroup

CPU utilization

% of the server's cpu budget (its cpuset), averaged over each stage. 100% = fully using its allotted cores; a single-threaded server tops out at 1/N.

Memory — working set

memory.current minus reclaimable file cache, averaged over each stage. This is what docker/k8s report as memory usage.

Network throughput is the one saturation signal still not sampled.

Data source: embedded snapshot. This page fetches bench/loadgen/combined.json at load time and re-renders from it; when fetch is blocked (opened as a file://, or the read-only hosted artifact) it falls back to the embedded snapshot. To auto-load your latest run, serve the repo over http — e.g. python3 -m http.server from the repo root, then open /bench/deploy/redis-vs-memcached.html.

Methodology. Four backends via bench/run-all-isolated.sh: every server and the client in its own Docker container (cgroup v2), servers on cpuset 0-7 (8 cpus, 4 GB) and the client on 8-19, official/native clients (pooled, HZ_CONNS=128), 4 s/stage. BonsaiGrid runs containerized too (glibc-matched image, seccomp=unconfined for io_uring). The dashed BonsaiGrid line is the native raw-protocol driver (bench ladder, from results-bonsaigrid-fair.json) — a client-isolated view of the same server. CPU and memory are sampled from each server's cgroup v2 (cpu.stat / memory.current) every 250 ms and averaged per stage; network throughput is not sampled.
