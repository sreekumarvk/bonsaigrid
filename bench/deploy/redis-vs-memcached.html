A fair run: every server and the load client live in their own cgroup on disjoint CPUs (servers on cpuset 0-7, client on 8-19), each server capped at 8 cpus / 4 GB, so the client can't steal cores from the server under test. Same official Hazelcast client drives both, so BonsaiGrid's faster server shows through once CPUs aren't contended. Why this run is fair — and what changed

The isolation. Each server and the load client run in a separate cgroup v2 with a disjoint cpuset — servers on cpus 0-7, the client on 8-19 — plus an equal 8-cpu / 4 GB cap per server. In the earlier co-located run the CPU-heavy Hazelcast Go client and the server shared cores, so throughput measured the Linux scheduler, not the server.

The reversal. Co-located, BonsaiGrid read 74k and lost to Hazelcast's 150k. Isolated, it reads 160k and beats Hazelcast (137k). Both are driven by the same official Hazelcast client, so this is the faster server showing through once it isn't starved. Memcached (348k) still leads on raw throughput — a thin binary protocol and a multithreaded server — which the heavyweight Hazelcast client can't match on the BonsaiGrid/Hazelcast side.

Server ceiling. BonsaiGrid's native raw-protocol driver (bench ladder, dashed) drives the same 8-core server to ~425k ops/s — so the remaining gap to Memcached is the official-client tax, not the store. A thin uniform client per protocol (RESP / memcached text / raw Hazelcast frames) would close it.

Methodology. Four backends via bench/run-all-isolated.sh: every server and the client in its own Docker container (cgroup v2), servers on cpuset 0-7 (8 cpus, 4 GB) and the client on 8-19, official/native clients (pooled, HZ_CONNS=128), 4 s/stage. BonsaiGrid runs containerized too (glibc-matched image, seccomp=unconfined for io_uring). The dashed BonsaiGrid line is the native raw-protocol driver (bench ladder, from results-bonsaigrid-fair.json) — a client-isolated view of the same server. CPU/memory/network panels are intentionally empty — not sampled, not fabricated.
