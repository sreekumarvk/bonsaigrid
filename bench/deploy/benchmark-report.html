Method & fairness

  • One client, swap the backend. A single Go program drives every backend with the same SET+GET of a ~90-byte JSON value keyed by UUID. The two Hazelcast-protocol targets — BonsaiGrid and Hazelcast — use the identical hazelcast-go-client code, only the host differs; Redis and Memcached use their native clients.
  • Staged closed-loop ramp. Concurrency steps 1 → 128 (geometric); each stage runs 4 s after a 2 s warm-up. Latency is timed client-side around each op and reduced to p50/p90/p99 per stage.
  • Single node, no backups, no persistence for every backend, to match single-instance Redis/Memcached. BonsaiGrid’s durability/backup/CP lanes are measured separately.
  • Caveats. All processes share one host over loopback, so high-concurrency numbers converge toward a client/loopback ceiling. Redis is single-threaded and Memcached has no partitioning or replication, so this is what a client experiences at parity config — not an architecturally identical comparison. BonsaiGrid and Hazelcast do strictly more (partitioning, sync backups, listeners, SQL).
  • Notable: this run also confirmed hazelcast-go-client conformance against BonsaiGrid (connect, cluster view, SetWithTTL/Get) — the Go client had not previously been exercised.
Reproduce: cargo build -p server --release && ./target/release/server, start the redis / memcached / hazelcast containers, then TARGET=<backend> go run ./bench/loadgen.
